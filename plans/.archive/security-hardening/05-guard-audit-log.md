## Goal
Every guard-enforced deny (write-scope ACL, secret-read, network-egress, git-command) appends one line to `.agents/audit.log` (timestamp, agent/source, truncated command or path), so denials are visible after the fact without needing to reproduce them.

- `grep -c "" .agents/audit.log` matches the number of deny events triggered during verification, no more, no less.

## Notes / risks
- **This is the phase most likely to hit the Windows path-normalization bug** discovered during this plan's own authoring (`checkWritePermission`'s `rel` computation only strips POSIX-style `/`-prefixed absolute paths — see backlog/0011). If `dev` hits denies-when-it-should-allow on Windows while smoke-testing this phase, that's the same pre-existing bug, not a regression from this phase — fix it in `scripts/agent-guard-core.mjs` (and its scaffold-template twin) as a prerequisite, not as part of this phase's scope, unless the user asks to fold it in.
- Keep the audit-log helper append-only and side-effect-light — do not turn it into a general logging framework (rotation, levels, etc. are all out of scope; it's one line per deny, nothing more).
- `.agents/audit.log` growing unbounded over a long-lived project is a known, accepted trade-off — not addressed here (rotation would be speculative scope per the "no configurability that wasn't requested" rule).

## Resolution (2026-07-04)
- Added a standalone `scripts/audit-log.mjs` (not inside `agent-guard-core.mjs`, which has a documented "no fs side effects" constraint) exporting `appendAuditLog(entry)`. It appends `<ISO timestamp> | <agent_type or "unknown"> | deny | <reason, truncated to 150 chars>` to `.agents/audit.log`, creates the parent directory on first write via `mkdirSync(..., { recursive: true })`, and wraps everything in try/catch so a logging failure can never break the guard that calls it. A comment in the module documents the hard limitation: Claude's native `permissions.deny`/`permissions.ask`/`sandbox.filesystem.denyRead` are enforced inside Claude Code itself with no hook firing, so those denials never reach this log — only guard-script-mediated denials do.
- Wired `appendAuditLog()` into the deny branches of all three guard scripts, dogfood and scaffold-template twin identically:
  - `.claude/scripts/agent-guard.mjs` / `agentGuardMjsTemplate()` — one call per deny branch (read-only-agent denial, scope-violation denial), passing the real `agentType`.
  - `.codex/scripts/agent-guard-codex.mjs` / `agentGuardCodexMjsTemplate()` — one call in the single deny branch, passing the real `agentType` (recovered from the SubagentStart temp file).
  - `.codex/scripts/codex-permission-guard.mjs` / `codexPermissionGuardMjsTemplate()` — one call before the combined `process.stdout.write` (covers all three pattern groups: git / secret-read / network-egress), passing `agentType: 'unknown'` since Codex PreToolUse payloads carry no agent identity field for this hook (confirmed in `.agents/memory/dev/MEMORY.md`).
  - Added `scripts/audit-log.mjs` to the SHARED bucket of `buildClaudeFileMap` in `src/scaffold/shared/claude-setup.ts` (imported by both harnesses' adapters, same as `agent-guard-core.mjs`).
- **Gitignore decision**: `.agents/audit.log` is local-only noise (append-only diagnostic log, not something a team needs to review in git history) — added to both `src/scaffold/react-vite/templates/gitignore.ts` (scaffolded projects) and this repo's own `.gitignore`. Chrome-extension scaffold reuses the same template per the phase 01 note, so only one file needed the edit.
- Confirmed via a throwaway `scripts/tmp-render-audit-log.ts` (deleted after use) that `buildClaudeFileMap({ harness: 'both', ... })` emits `scripts/audit-log.mjs` with the expected content, and that all three emitted guard scripts import `audit-log.mjs` and call `appendAuditLog(...)`. A second throwaway diff script confirmed the emitted `.codex/scripts/agent-guard-codex.mjs` and `.codex/scripts/codex-permission-guard.mjs` are logic-identical to their dogfood copies (the only differences are pre-existing dogfood-only comments predating this phase, e.g. the "LIVE VERIFICATION STATUS" block and cross-reference lines — not something this phase introduced).
- Verify: all four phase-listed commands produced the expected results — `agent-guard.mjs` deny for `planner` writing `src/index.ts` added a line referencing `planner`; `codex-permission-guard.mjs` deny for `cat .env` added a line referencing the truncated command; `agent-guard.mjs` allow for `planner` writing `plans/x.md` added no line; re-ran all phase 03/04 regression commands (`grep API_KEY .env.production`, `printenv`, `cat README.md` silent, `env FOO=bar cmd` silent, `git push`, `curl ... -d @.env`, `wget ...`) — all denies/silents unchanged, and exactly one audit-log line was added per new deny (5 new lines for the 5 regression denies, 0 for the 2 silent/allow cases). `npx tsc --noEmit` and `npm run build` both succeeded.
- Left `.agents/audit.log` in place after testing (not cleaned up) — it is gitignored (see decision above) and its content is useful as a working example of the format; the noise it contains from this session's own testing (several `dev`-agent denials triggered incidentally by scope-guard checks on my own tool calls, plus the deliberate smoke-test lines) is exactly the kind of local, non-committed diagnostic data the log is meant to hold.
