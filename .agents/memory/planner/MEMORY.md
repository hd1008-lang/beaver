# planner — Agent Memory

Short-term memory with a lifecycle — NOT an append-only log. One bullet per durable, non-obvious planning gotcha, newest first. Link related docs/ files. Read this file at the start of every session. Budget ≤ 15 bullets / ≤ 100 lines; over budget → run the beaver-memory-retro skill (dedupe, delete stale, promote durable facts to docs/).

- **`interpolate()` throws on leftover `{{tokens}}` in the OUTPUT (assets.ts) — asset token changes and renderer changes must land in the SAME phase** — a re-tokenized `harness-assets/` file with an old call site (or vice versa) fails every render/golden/snapshot test, so plans cannot split "author canonical asset" and "renderer split" into separate green phases; new UNconsumed assets are the only safe phase-01 authoring. Also: substituted VALUES are scanned by the leftover check — `{{x}}`-shaped text inside a param string throws at scaffold time. See `plans/.archive/neutral-canonical-harness/`.
- **tsup bundling flattens `import.meta.url` — asset resolution must walk up, not use a fixed relative path** — after tsup, every module's `import.meta.url` is `dist/index.js`, but under `tsx src/index.ts` it is the source file's own path. Any runtime asset loader (`harness-assets/`) must probe upward for the assets dir instead of hardcoding `../harness-assets`. See `plans/.archive/assets-and-tests/01-packaging-investigation.md`.
- **Byte-identical render diff is the safety rail for template-literal → file extractions** — when moving templates out of `harness-setup.ts`, capture a BEFORE render of `buildHarnessFileMap` (fixed params, all harness modes × testing on/off) and diff AFTER. Classic failure: editor-added trailing newline. Also pin `harness-assets/** text eol=lf` in `.gitattributes` — checked-in assets are subject to autocrlf; template literals never were.
- **Planner cannot write `backlog/` (guard hard-blocks; confirmed live 2026-07-05)** — when a plan needs a `backlog/NNNN-*.md` entry, record the verbatim content/back-link line in `00-overview.md` as a "first touch" dev task and delegate creation to dev in a phase step.
- **Harness-change plans: dogfood-first, scaffold-second, spec-handoff-third** — split cross-cutting harness moves into (1) dogfood repo changes, (2) scaffold template/asset changes, (3) spec/backlog handoff — never mix dogfood and template edits in one phase. Every plan touching `harness-assets/` or `harness-setup.ts` must end with a regen step: `npx tsx test/helpers/regen-dogfood.ts` (the golden test enforces zero drift).
- **Agent registry: `test-writer` stays outside `AGENTS`** — it is cart-conditional (only when `params.testing` is set). The `AGENTS` table covers only the five unconditional core agents; guard/validator merge a local `TEST_WRITER_DEF` at call site inside `buildHarnessFileMap`.
- **Agent tool lists are NOT purely derivable from `writeScope`** — advisor and scout both have empty writeScope but different tool sets (advisor has Bash/WebFetch/WebSearch; scout does not). Handled by an internal `TOOL_OVERRIDES` map — a template-implementation detail, NOT part of the 4-field registry schema.
- **Harness-only skeleton AGENTS.md stubs MUST include the Agent Routing table + PARK RULE** (post-0016 the canonical doc is AGENTS.md; the shared asset guarantees both sections) — stubs are minimal only for conventions/stack details, not agent workflow sections. (An earlier memory note said "never add" — that was wrong; corrected in `plans/.archive/skeleton-agent-routing/`.)
- **Spec-gap handling: never block a plan on a missing/incomplete spec** — infra-only work with no `docs/features/` spec is planner-executable on the brief alone; flag gaps as `backlog/NNNN-*-spec.md` entries for docs-writer instead of stopping. Converse rule: any "FLAG for docs-writer" or non-blocking follow-up prose in a plan MUST become a backlog entry BEFORE the plan is archived — prose in archived plans is invisible to future agents.
- **Park-rule wording is cart-independent** — the "## Park rule (anti-loop)" section in dev agent templates is identical across all project types. Keep it as literal string duplication; indirection for one paragraph costs more than it saves.
- **Read `AM` (added+modified) files before editing** — git status `AM` means added to the index then modified in the working tree; content may not match what was staged. Never assume content from git status alone.
