# planner — Agent Memory

Append durable, non-obvious planning gotchas and patterns discovered while working — one bullet each, newest first. Link related docs/ files. Read this file at the start of every session.

- **`.agents/memory/` seeds belong in `shared`, not `claudeOnly`** — they are harness-neutral; Codex TOMLs reference them too. The four core agents (dev, docs-writer, planner, advisor) must be in the unconditional `shared` FileMap block. `test-writer` stays in the `wantClaude` branch (no `.codex/agents/test-writer.toml` exists). No dedup guard needed for `both` mode if seeds are in `shared` only. See `plans/codex-memory-seed/`.
- **Neutral memory path: dogfood-first, scaffold-second, spec-handoff-third** — when moving agent memory to a harness-neutral path, split into three phases: (1) move real files + update agent definition files + update guard core in the dogfood repo (all in one phase so no agent is left reading from one place and writing to another); (2) update scaffold templates in `src/scaffold/`; (3) file spec backlog entry + archive. Do NOT mix dogfood edits and template edits in the same phase. See `plans/memory-neutral-path/`.
- **Codex harness port: verify hook payload schema BEFORE writing the guard adapter** — Codex docs don't name the PreToolUse payload fields for subagent identity / file path. Phase 01 of `plans/codex-harness-port/` must confirm these (ideally by echoing stdin from a live hook) before phase 02 writes the adapter. Silent fail-open is the risk. See `plans/codex-harness-port/01-verify-codex-hook-payload.md`.
- **Dual-harness single-source strategy: symlinks for dogfood, real files for scaffold output** — the beaver repo (dogfood) can use WSL2 symlinks for skills/memory so `.agents/` and `.codex/` point into `.claude/` without duplication. Scaffolded end-user projects cannot use symlinks (they must emit real files at both paths). Keep these two strategies separate in the plan. See `plans/codex-harness-port/03-skills-and-memory-symlinks.md` and `07-scaffold-codex-phase-b.md`.
- **No codex-harness spec exists yet — plan is not blocked on it** — the request is dogfood harness work (WHAT is fully defined by the user brief), not a new scaffold feature requiring a spec first. Flag the spec gap as a backlog entry for docs-writer (backlog/0005) rather than stopping. See `plans/codex-harness-port/07-scaffold-codex-phase-b.md`.
- **Agent definition files are the one unavoidable duplication across harnesses** — Claude uses MD+YAML frontmatter; Codex uses TOML. Two parsers, two formats, 5 files each. Do not code-gen; write by hand. Behavioral content must stay in sync manually; add a comment reminder in each format. See `plans/codex-harness-port/04-codex-subagent-tomls.md`.
- **`validatePlansMjsTemplate` is static (no params)** — unlike `validateStructureMjsTemplate` which bakes in the AGENTS registry, `validate-plans.mjs` references only the `plans/` and `backlog/` directory names (constant across project types). No parameterization needed; copy verbatim from the dogfood reference at `scripts/validate-plans.mjs`. See `plans/governance-propagate/01-validate-plans-template.md`.
- **`backlog/` writes are dev scope; planner cannot file backlog entries** — when a plan's overview needs a `backlog/NNNN-*.md` entry created, record the required content in the plan (overview or phase file) and delegate creation to dev in a phase step. The guard hard-blocks planner writes to `backlog/`. See `plans/governance-propagate/04-sync-dogfood.md`.
- **Agent registry: `test-writer` stays outside `AGENTS`** — it is cart-conditional (only emitted when `params.testing` is set). The `AGENTS` table covers only the five unconditional core agents. When the guard or validator need test-writer's scope, build a local `TEST_WRITER_DEF` constant and merge it at call site inside `buildClaudeFileMap`. See `plans/agent-registry/03-registry-guard.md`.
- **Agent tool lists are NOT purely derivable from `writeScope`** — advisor (empty writeScope) and scout (empty writeScope) have different tool sets (advisor has Bash/WebFetch/WebSearch; scout does not). Use an internal `toolOverride?: string[]` fifth field on `AgentDef` to handle exceptions. Do NOT add it to the 4-field spec schema — keep it a template-implementation detail. See `plans/agent-registry/02-derive-agent-files.md`.
- **Guard rename creates a brief dogfood window** — renaming `planner-guard.mjs` → `agent-guard.mjs` in the template (phase 03) before updating `.claude/settings.json` in the beaver repo (phase 05) leaves the repo's PreToolUse hook referencing a missing file. Not load-bearing for builds, but execute phases 03 and 05 in the same session when possible to avoid confusion. See `plans/agent-registry/`.
- **Dogfood sync is a distinct final phase** — template changes in `src/scaffold/shared/claude-setup.ts` do NOT automatically update the beaver repo's own `.claude/` directory. Every plan that touches the harness template must include a final "sync dogfood" phase that renders the new output and copies it into `.claude/`. See `plans/agent-registry/05-dogfood-and-beaver-sync.md`.
- **Harness-only skeleton CLAUDE.md stubs MUST include Agent Routing table + PARK RULE** — `claudeHarnessTableTemplate()` is called in all three `*-skeleton.ts` `claudeMdTemplate` functions, same as full scaffold. The stubs are minimal only for conventions/stack details, not for agent workflow sections. The spec (`docs/features/claude-harness/claude-harness.spec.en.md`) explicitly lists "harness-only skeletons" as a target for these sections. (Earlier memory note said "never add" — that was wrong; corrected in `plans/skeleton-agent-routing/`.)
- **`backlog/README.md` belongs in shared, not per-type** — it is project-agnostic like `plans/README.md`. Add it once in `buildClaudeFileMap` in `src/scaffold/shared/claude-setup.ts`, no cart conditioning needed.
- **claude-harness spec gaps surface during propagation plans** — `docs/features/claude-harness/claude-harness.spec.en.md` does not describe what `buildClaudeFileMap` emits in the "Related Files" list with enough detail. When the harness grows, flag the spec for docs-writer to update in parallel; do not block the plan on it.
- **Park-rule wording is cart-independent** — the "## Park rule (anti-loop)" section in dev agent templates is identical across all project types. Keep it as a literal string duplication rather than a shared helper — adding indirection for one paragraph costs more than it saves. See `docs/architecture/agent-workflow.en.md`.
- **Spec-gap follow-ups must become backlog entries before a plan is archived** — if a plan's `00-overview.md` contains a "Spec gap — FLAG for docs-writer" (or any non-blocking follow-up), convert it to `backlog/NNNN-<slug>.md` BEFORE archiving or marking the plan done. Prose in archived plans is invisible to future agents. See `plans/governance-cleanup/01-anchor-backlog-entry.md`.
- **Done plans accumulate silently — archive as part of the closing step** — once all phases are `done`, the plan should be moved to `plans/.archive/<slug>/` in the same work session. Add an explicit archive step to every plan's last phase (or as a dedicated final phase). See `plans/governance-cleanup/02-archive-done-plans.md` and `plans/README.md:73`.
- **validate-plans.mjs is the plans/backlog equivalent of lint-docs-frontmatter.mjs** — run `node scripts/validate-plans.mjs` after any plan or backlog change to surface table/frontmatter drift, all-done plans not archived, backlog ID gaps, and broken two-way links. Added in `plans/governance-cleanup/03-validate-plans-script.md`.
- **Read `AM` (added+modified) files before editing** — git status `AM` means the file was added to the index and then modified in the working tree. These are new files that may not match what git staged. Always read them before editing; never assume their content from git status alone.
- **`productDescription` is for CLAUDE.md rendering only — `buildClaudeFileMap` body unchanged** — when adding fields to `ClaudeHarnessParams`, check whether `buildClaudeFileMap`'s body actually needs to use the field. For `productDescription`, the field is consumed only by each project type's `claudeMdTemplate`, not by any shared emitted file. Adding it to the interface (phase 01) and the caller's params object (phases 03–04) is sufficient; no change to the function body, no `.claude/` content drift in dogfood sync. See `plans/harness-product-description/`.
- **Harness-only skeletons: add `## Project Overview` section for productDescription** — the skeleton CLAUDE.md stubs have no `## Project Overview` section. When rendering `productDescription` into them, insert the section after the opening `>` callout block and before `## Docs`. Keep it to one line (the description) — do not add stack lists, agent tables, or any other content (stubs are intentionally minimal). See `plans/harness-product-description/04-templates-skeleton.md`.
