- **Bash string interpolation with `$()`/backtick-heavy content is fragile in this shell**: passing a JS/TS template string containing literal backticks and `$HOME`/`$(...)`-shaped text through `node -e "..."` causes bash to interpret `$(...)` and backslash-escapes before node ever sees them (silently truncates or errors line-by-line). For any script body containing backticks, `$`, or code snippets, write a real `.mjs`/`.ts` file with the Write tool (under `test/`) and run it with `node file.mjs`, rather than inlining via `node -e`.

- **Harness conditionality (`harness` param in `ClaudeHarnessParams`)**: `buildClaudeFileMap` now takes `harness: 'claude' | 'codex' | 'both'` and splits the file map into three buckets (shared / claudeOnly / codexOnly). Callers map `cart.ai` → `harness` via a local `aiToHarness()` helper. Layout gates changed from `cart.ai === 'CLAUDE'` to `cart.ai !== 'NOT_USING'`. Bucket rule: classify by OWNERSHIP not by path prefix. SHARED: `scripts/*` (7 files — docs tooling, agent-guard-core.mjs, docs-first-reminder.sh). CLAUDE-ONLY: `.claude/scripts/agent-guard.mjs` (the ONLY .claude/scripts/ file), `.claude/settings.json`, `.claude/agents/*.md`, `.claude/agent-memory/`, `.claude/skills/`. CODEX-ONLY: `.codex/scripts/agent-guard-codex.mjs`, `.codex/scripts/codex-*.mjs`, `.codex/agents/*.toml`, `.codex/hooks.json`, `.agents/skills/` twins. The `ai` enum for react-vite, chrome-extension, and harness-only now all have four values (NOT_USING / CLAUDE / CODEX / BOTH); harness-only has its own `HARNESS_MENU_AI` constant (no NOT_USING — harness is required when choosing that flow).

- **Embedding computed regex fragments into a giant emitted-file template literal** (`claude-setup.ts`, e.g. `codexPermissionGuardMjsTemplate()`): when a template computes a string at TS/Node runtime (e.g. a regex-fragment builder that turns glob patterns into `\.env|\.pem` etc. using real single backslashes) and interpolates it via `${...}` into the giant outer template literal representing the emitted `.mjs` file, the computed value's backslashes pass through untouched — no extra escaping needed, because interpolation is a runtime string concat, not source-literal parsing. Only literal regex syntax written directly in the outer template source (e.g. `\b`, `\s`) needs `\\b`/`\\s` (double backslash) so it survives the outer template literal's own escape processing — otherwise `\b` becomes an actual backspace char in the emitted file. This is the same convention the pre-existing `DENY_PATTERNS` block in that file already used; verify via a throwaway tsx render (see `beaver-conventions` skill) rather than reasoning about escaping in your head.

- **`ClaudeHarnessParams` required fields for a throwaway render**: `buildClaudeFileMap()` needs `productDescription: string` and `seedDocs: FileMap` (an array — do not omit or cast loosely with `as any` on a partial object, since `params.seedDocs` is spread with `...params.seedDocs` and throws `TypeError: ... is not iterable` if missing/undefined). Minimal working call: `{ projectName, slug, flowEnum: string[], layerEnum: string[], harness, productDescription, conventionsSkill, seedDocs: [] }`.

- **Claude Code `permissions.ask` tier is real and documented** (verified 2026-07-04 via `https://code.claude.com/docs/en/permissions`, not just inference from schema shape): the permission system is `deny` → `ask` → `allow` in that evaluation order, all three are peer arrays under `permissions` in settings.json. Don't assume `ask` needs a fallback to `deny` without checking this page first — WebSearch summaries alone weren't enough to be fully confident; the official page's "Manage permissions" section has the authoritative sentence.

- **Live proof that the write-scope guard hook is wired to your own tool calls**: while implementing audit logging (plans/security-hardening phase 05), an Edit attempt to `.gitignore` (outside dev's writeScope) was denied by the harness's own PreToolUse hook, and that denial appended a real line to `.agents/audit.log` — before I'd run any smoke test. This is useful evidence when verifying guard wiring: your own blocked tool calls during the session are legitimate audit-log entries, not test noise: don't be surprised by extra lines when you `tail`/`wc -l` the log.
- **Three separate `.claude/scripts/agent-guard.mjs`-shaped templates exist and are NOT the same shape**: `agentGuardMjsTemplate()` (src/scaffold/shared/claude-setup.ts, ~line 420, self-contained WRITE_SCOPES + logic inline) is what actually gets emitted to `.claude/scripts/agent-guard.mjs` in scaffolded projects, while the *dogfood* copy at `.claude/scripts/agent-guard.mjs` in this repo imports `checkWritePermission` from `agent-guard-core.mjs` instead (matching `agentGuardCodexMjsTemplate()`'s shape, not `agentGuardMjsTemplate()`'s). This mismatch pre-dates phase 05 and is out of scope to fix — when editing "the agent-guard.mjs template," check which of the two shapes you're actually looking at before assuming dogfood and scaffold-template are structurally identical.
- **Both-mode (`tsx` vs compiled `dist`) byte-identical verification for `import.meta.url`-relative asset resolution**: don't try to import `buildClaudeFileMap` from the real `dist/index.js` — that bundle's only export is the CLI's interactive `main()`, it doesn't re-export internal functions. Instead bundle a *separate* throwaway entry file with `npx tsup <file>.ts --format esm --out-dir <tmp-dir>` (any out-dir under the project root works, since the walk-up logic in `resolveAssetsDir()` just needs `harness-assets/` to be a sibling of some ancestor), then `node <tmp-dir>/<file>.js` and compare SHA-256 against the dev-mode (`npx tsx`) output. Delete the throwaway `--out-dir` afterward.
- **`python3` alias on this Windows box is broken** (resolves to the Microsoft Store app-execution-alias stub, "Python was not found... install from the Microsoft Store"). Use `node` with a real `.mjs` script file instead (Write tool, under `test/`), run with `node test/tmp-*.mjs`, then delete it.
- **Extracting template-literal content into real asset files without hand-transcribing**: when a phase (e.g. assets-and-tests phase 02) requires moving ~13 zero-param template functions from a giant `claude-setup.ts`-shaped file into real files byte-for-byte, don't manually copy/paste template literal bodies — re-render `buildClaudeFileMap()` BEFORE any edits (throwaway tsx script → JSON array of `{relativePath, content}` sorted and dumped to a temp file), then a second throwaway node script reads that JSON and `writeFileSync`s each target's `content` straight to its `harness-assets/` path. This sidesteps all backtick/`${}`-escaping transcription risk entirely, and the same BEFORE JSON doubles as the byte-identical baseline for the final `git diff --no-index` check. For deleting the now-orphaned template functions afterward (giant multi-hundred-line blocks with heavy backtick/`${}` content that make Edit's `old_string` matching fragile), use a small node script that slices the file by 1-indexed line numbers with a sanity-check assertion on the first/last line content before mutating — cheaper and more reliable than one giant Edit call once you've located the boundaries via Grep.
- **Extracting TOKENIZED templates (phase 03, `{{token}}` placeholders) doesn't need the sentinel-render dance**: since the asset content with placeholders is never actually rendered as-is by the real code (real renders always substitute real values), the safest path is: read the exact original template-function source (already in context from a prior Read), hand-copy it into a scratch file via the Write tool at an in-scope path (Write's `content` param is literal final bytes, so no escaping/backtick risk at all, unlike embedding into ANOTHER template literal), replacing `${paramExpr}` with `{{tokenName}}` at each dynamic point. Verify correctness after the fact with a full BEFORE/AFTER byte-identical re-render diff (6 combos: claude/codex/both × testing on/off) rather than trying to verify the placeholder substitution in isolation.
- **`interpolate(content, tokens)` token pattern must be word-chars-only (`\{\{[a-zA-Z0-9_]+\}\}`), not a bare `\{\{.*?\}\}`**: `harness-assets/scripts/agent-guard-core.mjs` contains a legitimate literal JSDoc type annotation `{{ decision: 'allow' | 'deny' | 'pass', reason?: string }}` (spaces/colons/quotes inside the braces) that must pass through untouched. A loose "any `{{...}}`" leftover-check would false-positive on this and throw even after all real tokens were correctly substituted. Before tokenizing any asset, grep the whole `claude-setup.ts` source for literal `{{` to rule out this class of collision (found exactly one, in the JSDoc above) — see `plans/assets-and-tests/03-tokenized-assets.md` Resolution section for the full writeup.
- **Testing a tokenized asset (`harness-assets/scripts/agent-guard-core.mjs`, has `{{writeScopesJson}}`) requires interpolating BEFORE import/spawn — the raw file is not valid JS on its own** (`export const WRITE_SCOPES = {{writeScopesJson}};` is a syntax error). Pattern used in `plans/assets-and-tests/06-port-security-tests.md`: a `test/helpers/prepare-guard-dir.ts` helper builds a throwaway `os.tmpdir()` directory mirroring the real relative-import layout (`scripts/`, `.claude/scripts/`, `.codex/scripts/`), writes `interpolate(readAsset('scripts/agent-guard-core.mjs'), { writeScopesJson })` (scopes built from the exported `AGENTS` registry in `claude-setup.ts`, same as `writeScopesJson()` does internally) plus verbatim copies of the untokenized adapter/audit-log assets, then tests `import()`/spawn against that temp dir — never the dogfood copies (per the phase's explicit instruction, since the golden test already proves dogfood == rendered asset). `checkWritePermission` itself has no fs side effects, so it's a plain dynamic import; the adapters (`agent-guard.mjs`, `codex-permission-guard.mjs`) need `child_process.spawnSync` with stdin JSON (no shell `echo`, works on Windows) and an explicit throwaway `cwd` option, since `audit-log.mjs` resolves `.agents/audit.log` as a path relative to the process cwd, not `payload.cwd` — passing only a JSON `cwd` field in the payload does nothing; the actual OS-level `cwd` spawn option is what matters.
- **Rendering `buildClaudeFileMap({ harness: 'both', testing: {...} })` produces a self-contradicting `scripts/validate-structure.mjs`**: the emitted validator's "primary owned directory" uniqueness check compares each agent's `writeScope[0]`, and `dev` (`['src/', 'test/', ...]`) collides with `TEST_WRITER_DEF` (`['src/', 'test/', 'tests/', 'e2e/', 'playwright/']`) — both start with `'src/'`. Any test that spawns the emitted validate-structure.mjs against its own `both`+`testing` render must either render WITHOUT `testing` (workaround used in `test/validator-selftest.test.ts`) or expect that specific failure. Filed as `backlog/0018-testwriter-devscope-collision.md` — real fix (reorder `TEST_WRITER_DEF.writeScope` so `test/` is first, or relax the uniqueness check) is out of scope until a dedicated phase touches `claude-setup.ts`'s agent-scope model.

- **`scaffoldReactVite(cart)` resolves `cart.projectName` relative to `process.cwd()`, not as a path** (`path.resolve(process.cwd(), projectName)` in `src/scaffold/react-vite/index.ts`) — passing an absolute path as `projectName` breaks because nested slug-derived paths (e.g. `.claude/skills/${projectName}-conventions/`) get built from the raw string, producing a mangled path like `.claude\skills\c:\users\...-conventions`. For a non-interactive/CI scaffold entry point (`test/helpers/scaffold-fixture.ts`), `process.chdir()` into the target's parent dir first, then pass only `path.basename(targetPath)` as `projectName` — this mirrors how the interactive CLI is actually invoked (user picks a project name while already `cd`'d into the parent).
- **`npx tsx <file>.ts` DOES resolve `@src/*`/`@utils/*` path aliases from `tsconfig.json`** (no separate loader config needed) — confirmed 2026-07-05 running `test/helpers/scaffold-fixture.ts` standalone (not through vitest, which has its own `resolve.alias` in `vitest.config.ts`). Don't assume alias resolution requires the vitest config; tsx's own tsconfig-paths support covers ad-hoc scripts too.
