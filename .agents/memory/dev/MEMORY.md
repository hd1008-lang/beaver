- **`.agents/memory/<agent>/MEMORY.md` seed placement**: The four core-agent seeds (dev, docs-writer, planner, advisor) live in the `shared` block of `buildHarnessFileMap` ‚Äî NOT `claudeOnly`. `test-writer` seed stays in the `wantClaude` branch (no `.codex/agents/test-writer.toml` exists). - **Throwaway verification scripts go under `test/`**: the scratchpad/OS-temp dir and top-level `scripts/` are outside dev's writeScope (the guard checks absolute paths, not just repo-relative ones). Write to `test/tmp-*.mjs`/`.ts` (in scope), run with `node`/`npx tsx`, then delete via Bash `rm`.

- **Bash string interpolation with `$()`/backtick-heavy content is fragile in this shell**: passing a JS/TS template string containing literal backticks and `$(...)`-shaped text through `node -e "..."` lets bash interpret `$(...)` and backslash-escapes before node sees them (silent truncation or line-by-line errors). Write a real `.mjs`/`.ts` file with the Write tool (under `test/`) and run it with `node`, rather than inlining via `node -e`.

- **Harness conditionality (`harness` param)**: `buildHarnessFileMap` takes `harness: 'claude' | 'codex' | 'both'` and splits the file map into shared / claudeOnly / codexOnly buckets ‚Äî classify by OWNERSHIP, not path prefix (bucket contents documented in the ai-harness spec's output tables). Callers map `cart.ai` ‚Üí `harness` via a local `aiToHarness()` helper; layout gates use `cart.ai !== 'NOT_USING'`. The `ai` enum has four values (NOT_USING / CLAUDE / CODEX / BOTH); harness-only uses its own `HARNESS_MENU_AI` constant without NOT_USING.

- **Embedding computed regex fragments into a giant emitted-file template literal** (`harness-setup.ts`): values interpolated via `${...}` at runtime pass their backslashes through untouched ‚Äî no extra escaping. Only literal regex syntax written directly in the outer template source (e.g. `\b`, `\s`) needs `\\b`/`\\s` so it survives the template literal's own escape processing. Verify via a throwaway tsx render rather than reasoning about escaping in your head.

- **`HarnessParams` required fields for a throwaway render**: `buildHarnessFileMap()` needs `productDescription: string` and `seedDocs: FileMap` (an array ‚Äî `...params.seedDocs` throws `TypeError: not iterable` if missing). Post-0016 also required: `projectSections`, `extraRoutingRows` (interpolated into the AGENTS.md asset ‚Äî missing tokens throw at render). Minimal call: `{ projectName, slug, flowEnum: string[], layerEnum: string[], harness, productDescription, reminderTrigger, projectSections, extraRoutingRows, conventionsSkill, devAgent, seedDocs: [] }`.

- **Three separate `agent-guard.mjs`-shaped templates exist and are NOT the same shape**: `agentGuardMjsTemplate()` in `src/scaffold/shared/harness-setup.ts` (self-contained WRITE_SCOPES inline) is what scaffolded projects get at `.claude/scripts/agent-guard.mjs`, while this repo's dogfood copy imports `checkWritePermission` from `agent-guard-core.mjs` (matching the Codex adapter's shape). When editing "the agent-guard template," check which shape you're looking at first.

- **`scaffoldReactVite(cart)` resolves `cart.projectName` relative to `process.cwd()`, not as a path** ‚Äî passing an absolute path breaks slug-derived nested paths (e.g. `.claude/skills/${projectName}-conventions/`). For non-interactive/CI entry points (`test/helpers/scaffold-fixture.ts`), `process.chdir()` into the target's parent first and pass only `path.basename(targetPath)` as `projectName`.

- **`writeScope[0]` is a validator heuristic, not the real permission check** (backlog/0018, resolved 2026-07-11): `agent-guard-core.mjs`'s `checkWritePermission()` does a full array `.some(prefix => rel.startsWith(prefix))` membership check ‚Äî order-independent. Only `validate-structure.mjs`'s "primary owned directory uniqueness" check cares about `writeScope[0]`. When two agents legitimately share a directory (e.g. `dev` and `test-writer` both write under `src/`), reorder the array so each agent's *unique* dir comes first rather than changing the guard.

- **`sandbox.credentials` (Claude Code, requires v2.1.187+)** is a dedicated block, separate from `sandbox.filesystem.denyRead`: a `deny` entry both blocks the file read (like `denyRead`) AND unsets the named env var before every sandboxed Bash command ‚Äî `denyRead` alone does nothing about a credential already sitting in the process environment. Env vars also accept `mode: "mask"` (per-session sentinel, needs `network.tlsTerminate`, ignored from project-level settings). `harness-assets/.claude/settings.json` is the single asset (`readAsset()`) behind both the dogfood copy (regenerate via `npx tsx test/helpers/regen-dogfood.ts`) and scaffolded output ‚Äî edit once. Verified against live docs via WebFetch on `https://code.claude.com/docs/en/sandboxing` 2026-07-11. Related: `permissions.ask`/`deny`/`allow` tier (evaluation order `deny` ‚Üí `ask` ‚Üí `allow`) is the general permission-prompt mechanism, documented at `https://code.claude.com/docs/en/permissions` ‚Äî a separate, coarser layer from `sandbox.credentials`.
