- **validate-plans.mjs table parser**: `parseOrderedPhasesTable()` must scope parsing to the "## Ordered phases" section — plans/overview files often have multiple Markdown tables (e.g. "Owner map") whose rows also match `/^\|\s*\d+\s*\|/`. Use a section regex first, then parse rows within it.

- **Throwaway verification scripts go under `test/`**: the scratchpad/OS-temp dir and top-level `scripts/` are outside dev's writeScope (the guard checks absolute paths, not just repo-relative ones). Write to `test/tmp-*.mjs`/`.ts` (in scope), run with `node`/`npx tsx`, then delete via Bash `rm`.

- **Bash string interpolation with `$()`/backtick-heavy content is fragile in this shell**: passing a JS/TS template string containing literal backticks and `$(...)`-shaped text through `node -e "..."` lets bash interpret `$(...)` and backslash-escapes before node sees them (silent truncation or line-by-line errors). Write a real `.mjs`/`.ts` file with the Write tool (under `test/`) and run it with `node`, rather than inlining via `node -e`.

- **Harness conditionality (`harness` param)**: `buildHarnessFileMap` takes `harness: 'claude' | 'codex' | 'both'` and splits the file map into shared / claudeOnly / codexOnly buckets — classify by OWNERSHIP, not path prefix (bucket contents documented in the ai-harness spec's output tables). Callers map `cart.ai` → `harness` via a local `aiToHarness()` helper; layout gates use `cart.ai !== 'NOT_USING'`. The `ai` enum has four values (NOT_USING / CLAUDE / CODEX / BOTH); harness-only uses its own `HARNESS_MENU_AI` constant without NOT_USING.

- **Embedding computed regex fragments into a giant emitted-file template literal** (`harness-setup.ts`): values interpolated via `${...}` at runtime pass their backslashes through untouched — no extra escaping. Only literal regex syntax written directly in the outer template source (e.g. `\b`, `\s`) needs `\\b`/`\\s` so it survives the template literal's own escape processing. Verify via a throwaway tsx render rather than reasoning about escaping in your head.

- **`HarnessParams` required fields for a throwaway render**: `buildHarnessFileMap()` needs `productDescription: string` and `seedDocs: FileMap` (an array — `...params.seedDocs` throws `TypeError: not iterable` if missing). Post-0016 also required: `projectSections`, `extraRoutingRows` (interpolated into the AGENTS.md asset — missing tokens throw at render). Minimal call: `{ projectName, slug, flowEnum: string[], layerEnum: string[], harness, productDescription, reminderTrigger, projectSections, extraRoutingRows, conventionsSkill, devAgent, seedDocs: [] }`.

- **Three separate `agent-guard.mjs`-shaped templates exist and are NOT the same shape**: `agentGuardMjsTemplate()` in `src/scaffold/shared/harness-setup.ts` (self-contained WRITE_SCOPES inline) is what scaffolded projects get at `.claude/scripts/agent-guard.mjs`, while this repo's dogfood copy imports `checkWritePermission` from `agent-guard-core.mjs` (matching the Codex adapter's shape). When editing "the agent-guard template," check which shape you're looking at first.

- **`python3` on this Windows box is broken** (Microsoft Store app-execution-alias stub). Use `node` with a real `.mjs` script file (Write tool, under `test/`), run with `node test/tmp-*.mjs`, then delete it.

- **`scaffoldReactVite(cart)` resolves `cart.projectName` relative to `process.cwd()`, not as a path** — passing an absolute path breaks slug-derived nested paths (e.g. `.claude/skills/${projectName}-conventions/`). For non-interactive/CI entry points (`test/helpers/scaffold-fixture.ts`), `process.chdir()` into the target's parent first and pass only `path.basename(targetPath)` as `projectName`.

- **`npx tsx <file>.ts` DOES resolve `@src/*`/`@utils/*` path aliases from `tsconfig.json`** (no separate loader config; vitest separately uses `resolve.alias` in `vitest.config.ts`). Confirmed 2026-07-05 running `test/helpers/scaffold-fixture.ts` standalone.

- **Before executing a "pending" phase, check for already-written deliverables** (`git status`, `Glob` the target paths): plans/assets-and-tests phase 07 arrived fully pre-implemented from an interrupted session while its tracker still said `pending, 0/7`. Re-run the phase's own Verify steps first rather than re-implementing.

- **`writeScope[0]` is a validator heuristic, not the real permission check** (backlog/0018, resolved 2026-07-11): `agent-guard-core.mjs`'s `checkWritePermission()` does a full array `.some(prefix => rel.startsWith(prefix))` membership check — order-independent. Only `validate-structure.mjs`'s "primary owned directory uniqueness" check cares about `writeScope[0]`. When two agents legitimately share a directory (e.g. `dev` and `test-writer` both write under `src/`), reorder the array so each agent's *unique* dir comes first rather than changing the guard.

- **Interpolating `{{token}}` immediately followed by a literal separator is a leading/trailing-junk trap** (0017, `.beaver/` folder move): a template written as `{{baseDir}}/plans/` breaks when the token value is legitimately `''` (produces `/plans/`, leading slash). Fix: never interpolate a raw prefix value next to a hardcoded separator — pre-compute the *whole* joined string as its own token (e.g. `{{plansDir}}` = `baseDir ? baseDir+'/plans' : 'plans'`) so the empty case degrades cleanly. Same bug class applies to any `{{x}}/y` or `{{x}}.y` pattern.
- **A `baseDir`-style path prefix must be threaded through every consumer, not just the FileMap keys and prose**: relative `import`/`require` paths inside asset scripts that cross an immovable/movable boundary (e.g. `.claude/scripts/agent-guard.mjs`, root-pinned, importing `../../scripts/agent-guard-core.mjs`, which moves under baseDir) silently keep working syntactically but resolve to the wrong file once one side moves and the other doesn't — always grep asset scripts for `from '\.\./` and hardcoded bare dir constants (e.g. `DOCS_DIR = 'docs'` resolved via `process.cwd()`) before declaring a path-prefix refactor complete.
- **Test helpers that spread `beaverParams` as a generic fixture (not testing beaver's own dogfood) must explicitly override new HarnessParams fields with the scaffolded-project default**: `test/validator-selftest.test.ts` and `test/helpers/prepare-guard-dir.ts` both reused `{ ...beaverParams, ... }` to build "what a scaffolded project looks like" fixtures; adding `baseDir: '.beaver'` to `beaverParams` silently broke them until each call site got an explicit `baseDir: ''` / `scriptsDir: 'scripts'` override. Grep `grep -rn beaverParams test/` after any HarnessParams field addition.
- **`sandbox.credentials` (Claude Code, requires v2.1.187+)** is a dedicated block, separate from `sandbox.filesystem.denyRead`: a `deny` entry both blocks the file read (like `denyRead`) AND unsets the named env var before every sandboxed Bash command — `denyRead` alone does nothing about a credential already sitting in the process environment. Env vars also accept `mode: "mask"` (per-session sentinel, needs `network.tlsTerminate`, ignored from project-level settings). `harness-assets/.claude/settings.json` is the single asset (`readAsset()`) behind both the dogfood copy (regenerate via `npx tsx test/helpers/regen-dogfood.ts`) and scaffolded output — edit once. Verified against live docs via WebFetch on `https://code.claude.com/docs/en/sandboxing` 2026-07-11. Related: `permissions.ask`/`deny`/`allow` tier (evaluation order `deny` → `ask` → `allow`) is the general permission-prompt mechanism, documented at `https://code.claude.com/docs/en/permissions` — a separate, coarser layer from `sandbox.credentials`.
